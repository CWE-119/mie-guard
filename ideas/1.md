## 1. Introduction & Threat Model

The Intel Management Engine (ME) and AMD Platform Security Processor (PSP) are separate microcontrollers embedded in the Platform Controller Hub (PCH) or CPU die. They run a proprietary, closed‑source operating system with **full access to system memory, network interfaces, and peripherals**, completely bypassing the main OS and hypervisor.

A compromised ME/PSP can:

- Read or modify any RAM region (including cryptographic keys) via DMA.
- Send and receive network traffic through a **dedicated MAC address** invisible to the OS.
- Persist across reboots and OS reinstalls.
- Remain undetectable by any software running on the main CPU.

This document describes the construction of a **hardware‑based monitoring and mitigation platform** that operates **outside the trust boundary** of the target PC. It combines multiple independent sensors to detect and, optionally, block anomalous ME/PSP activity in real time.

## 3. Phase 1: The Network Gatekeeper (Lossless Packet Capture)

**Objective:** Detect any network communication originating from the Management Engine’s dedicated MAC address.

**Principle:** The ME has its own MAC address (typically `OS_MAC + 1` on Intel systems). It can bypass the OS network stack entirely. Placing a **transparent Ethernet bridge** between the PC and the LAN allows us to see every packet, regardless of the sender.

### Hardware Components

| Item | Model | Approx. Cost |
|------|-------|--------------|
| Single‑Board Computer | Raspberry Pi 4 (2GB+) | $35–$55 |
| Dual Gigabit Ethernet HAT | Waveshare CM4‑ETH‑ROUTER‑BOX or separate USB3‑GigE adapter | $20–$40 |
| MicroSD Card | 32GB+ | $10 |
| Power Supply | Official Raspberry Pi 5V/3A | $10 |
| Ethernet Cables | 2× CAT6 | $5 |

### Assembly

1. Connect the **Ethernet HAT** or USB adapter to the Pi. You should now have two network interfaces: `eth0` (built‑in) and `eth1` (external).
2. Insert the Pi between the **target PC** and your **LAN switch/router**:
   - `eth0` → Target PC
   - `eth1` → Upstream network

### Software Setup (Raspberry Pi OS Lite)

1. Install OS and enable SSH.
2. Update and install required packages:
   ```bash
   sudo apt update
   sudo apt install -y bridge-utils tshark suricata
   ```
3. Create a **transparent bridge**:
   ```bash
   sudo nmcli connection add type bridge ifname br0
   sudo nmcli connection add type ethernet slave-type bridge con-name bridge-eth0 ifname eth0 master br0
   sudo nmcli connection add type ethernet slave-type bridge con-name bridge-eth1 ifname eth1 master br0
   sudo nmcli connection modify br0 ipv4.method auto
   sudo nmcli connection up br0
   ```
4. Verify the bridge is working: `brctl show`.
5. Deploy a custom **Suricata rule** to alert on ME MAC activity. Create `/etc/suricata/rules/mei-guard.rules`:
   ```
   alert ether any any -> any any (msg:"ME MAC ADDRESS ACTIVE"; \
         eth.src:00:15:17; sid:1000001; rev:1;)
   alert ether any any -> any any (msg:"ME MAC ADDRESS ACTIVE"; \
         eth.src:3C:EC:EF; sid:1000002; rev:1;)
   ```
6. Enable Suricata in inline mode:
   ```bash
   sudo suricata -c /etc/suricata/suricata.yaml -q 0
   ```
7. (Optional) Forward logs to the central `mei-guard` analyser via syslog.

**Outcome:** Any packet with a source MAC belonging to Intel’s ME OUI (`00:15:17` or `3C:EC:EF`) triggers an immediate alert. This is the single most reliable indicator of remote ME activation.

## 4. Phase 2: The SPI Guard (Firmware Tampering Detection)

**Objective:** Detect unauthorised writes to the SPI flash chip containing the ME/PSP firmware.

**Principle:** The ME firmware is stored on an 8‑pin SPI NOR flash. By attaching a **passive sniffer** to the chip’s pins, we can monitor all read/write commands. The chip’s **Status Register** indicates when write protection is disabled—a necessary step before flashing malicious code.

### Hardware Components

| Item | Model | Approx. Cost |
|------|-------|--------------|
| USB‑to‑SPI Bridge | FTDI FT232H breakout (e.g., Adafruit 2264) | $15 |
| SOIC‑8 Test Clip | Pomona 5250 | $15 |
| Jumper Wires | Female‑to‑Female Dupont | $5 |

### Wiring Diagram

```
FT232H Pin          SOIC‑8 Clip Pin
──────────────────────────────────
ADBUS0 (SCK)   ───►  Pin 6 (CLK)
ADBUS1 (MOSI)  ───►  Pin 5 (SI)
ADBUS2 (MISO)  ───►  Pin 2 (SO)
ADBUS3 (CS)    ───►  Pin 1 (CS#)
GND            ───►  Pin 4 (GND)
+3.3V          ───►  Pin 8 (VCC)   *Only if the board is powered off*
```

**Warning:** Do **not** power the chip from the FTDI while the motherboard is powered on—this may damage both. For live monitoring, connect only GND, SCK, MISO, MOSI, CS; the chip receives power from the motherboard’s standby rail.

### Software Setup

On the monitoring PC (can be the same Pi from Phase 1):

1. Install `flashrom`:
   ```bash
   sudo apt install flashrom
   ```
2. Load the FTDI kernel module:
   ```bash
   sudo modprobe ftdi_sio
   echo 0403 6014 | sudo tee /sys/bus/usb-serial/drivers/ftdi_sio/new_id
   ```
3. Create a monitoring script (`spi_guard.py`) that:
   - Reads the **Status Register** (command `0x05`) every second.
   - Checks if the **Write Enable Latch (WEL)** bit (bit 1) is set.
   - If set and the system is not in BIOS update mode, triggers an alert.
   - Periodically reads the ME region of the flash and compares its SHA‑256 hash with a known‑good baseline.

**Sample `spi_guard.py` snippet:**
```python
import time
import subprocess
import hashlib

def read_status_register():
    out = subprocess.check_output(["flashrom", "-p", "ft2232_spi:type=232H", "--chip", "MX25L12835F", "-r", "/dev/stdout", "-c", "MX25L12835F", "-i", "0:1"])
    return out[0]

known_hash = "..."
while True:
    sr = read_status_register()
    if sr & 0x02:   # WEL bit
        print("ALERT: SPI Write Enable Latch set!")
        # Send alert via syslog or webhook
    time.sleep(1)
```

**Outcome:** Any attempt to modify the ME firmware (e.g., by an attacker flashing a backdoor) is detected in real time. Combined with Phase 1, this catches both remote and physical tampering.

## 5. Phase 3: The Power Oracle (Stealth Activity Detection)

**Objective:** Detect ME/PSP activity even when it does not generate network traffic or SPI writes (e.g., pure memory scraping).

**Principle:** The ME runs on the **3.3V Standby (3.3VSB)** rail, which is active even when the PC is "off". Its CPU and bus activity cause tiny fluctuations in current draw. By monitoring this rail with a high‑resolution current probe, we can identify the characteristic **32 kHz heartbeat** of the ME kernel and bursts of activity.

### Hardware Components

| Item | Model | Approx. Cost |
|------|-------|--------------|
| USB Oscilloscope / DAQ | Digilent Analog Discovery 3 | $300 |
| Current Probe | Digilent Current & Power Adapter (or Hantek CC‑65) | $70 |
| Hall Effect Current Sensor | ACS712 module (optional, for DIY) | $5 |

### Connection Method

**Option A (Non‑invasive):** Use a clamp‑on current probe around the **purple +5VSB wire** of the ATX 24‑pin connector. The 3.3VSB is derived from this rail, and ME activity modulates its current.

**Option B (More sensitive):** Identify the **3.3VSB pin** on the LPC debug header (often pin 8 or 9) and connect the current probe in‑line.

### Software Setup

1. Install WaveForms software (from Digilent) on the monitoring PC.
2. Configure the Analog Discovery 3 to sample the current channel at **≥ 1 MS/s**.
3. Export the data to a Python script for analysis.

**Detection Algorithm (Python with SciPy):**
```python
import numpy as np
from scipy import signal

def detect_me_activity(current_trace, sample_rate):
    # Look for 32 kHz harmonic (ME kernel clock)
    f, Pxx = signal.periodogram(current_trace, sample_rate)
    peak_32khz = np.max(Pxx[(f > 31000) & (f < 33000)])
    baseline = np.median(Pxx)
    
    # Also look for periodic bursts (DMA activity pattern)
    # ... (advanced: use CUSUM on RMS envelope)
    
    return peak_32khz > baseline * 10
```

**Integration:** The script can run on the same Pi (using `libsigrok` or `pydwf`) and send alerts to the central `mei-guard` analyser.

**Outcome:** Even a completely silent ME that never accesses the network or SPI flash is detectable via its power signature. This is a true **side‑channel defence**.

## 6. Phase 4: The PCIe Arbiter (Ultimate Control)

**Objective:** Actively block malicious DMA reads initiated by the PCH/ME.

**Principle:** By inserting an FPGA between the CPU and a critical peripheral (e.g., NVMe SSD or network card), we can filter all PCIe Transaction Layer Packets (TLPs). The FPGA can drop, modify, or log any memory read request originating from the PCH’s Bus:Device.Function (BDF).

### Hardware Components

| Item | Model | Approx. Cost |
|------|-------|--------------|
| M.2 FPGA Board | PicoEVB (Lattice ECP5) or NiteFury (Artix‑7) | $50–$200 |
| M.2 Extender / Riser | M.2 Key M to PCIe x4 adapter | $10 |
| JTAG Programmer | FT2232H Mini Module | $30 |

### Implementation Overview

1. **Install the FPGA board** in an M.2 slot (or use a riser to connect externally).
2. **Develop Verilog/VHDL code** using the open‑source **PCIe ARI** or **RIFFA** framework.
3. **Program the FPGA** to act as a **PCIe Protocol Analyzer**:
   - Sniff all **Memory Read Request (MRd)** TLPs.
   - Check the **Requester ID** field (bits [15:0]) against known PCH BDFs:
     - Intel: `00:1f.x`, `00:16.x`, `00:14.x`
     - AMD: `00:14.x`, `00:08.x`
   - If a match is found and the address falls outside whitelisted ranges (e.g., USB buffers), log the event and optionally **corrupt the completion data** or **drop the request**.

**Example Verilog Snippet (for LiteX):**
```verilog
always @(posedge clk) begin
    if (tlp_valid && tlp_type == MRd && requester_id inside {16'h001f, 16'h0016}) begin
        if (address[63:32] != 0 || address[31:0] > 32'h10000000) begin
            // Suspicious DMA – drop packet and raise interrupt
            drop_packet <= 1;
            alert <= 1;
        end
    end
end
```

**Alternative: Use existing open‑source projects** like **PCILeech** in monitor‑only mode, or **OpenCAPI** with custom filters.

**Outcome:** The ME’s ability to read arbitrary system memory via DMA is **neutered**. Even if the ME firmware is compromised, it cannot exfiltrate sensitive data.

## 7. Integration with `mei-guard` Software

All phases feed data into the **`mei-guard`** analysis daemon (described in the software documentation). The recommended setup:

- **Central Controller:** Raspberry Pi 4 running `mei-guard` (C daemon + Python analyser).
- **Phase 1:** Suricata logs are forwarded via `syslog` to the analyser.
- **Phase 2:** `spi_guard.py` writes JSON alerts to a Unix socket read by the analyser.
- **Phase 3:** The power analysis script sends JSON alerts via HTTP to the analyser’s webhook endpoint.
- **Phase 4:** The FPGA exposes a UART or I²C interface to the Pi; a small daemon translates FPGA interrupts into `mei-guard` alerts.

The web dashboard (`http://pi-ip:7473`) displays all alerts in a unified timeline.

## 9. Step‑by‑Step Assembly & Configuration

### 9.1 Phase 1 Setup

1. Flash Raspberry Pi OS Lite to microSD.
2. Boot Pi, run `sudo raspi-config` to enable SSH, set hostname.
3. Connect Ethernet HAT and cables as per diagram.
4. Configure transparent bridge (Section 3).
5. Install Suricata and add ME‑MAC rules.
6. Verify: unplug target PC LAN; Pi should still have internet.

### 9.2 Phase 2 Setup

1. Power off target PC.
2. Attach SOIC‑8 clip to BIOS chip (locate with motherboard schematic).
3. Connect FT232H to Pi USB port.
4. Install `flashrom` and test a read: `sudo flashrom -p ft2232_spi:type=232H -r backup.bin`.
5. Deploy `spi_guard.py` and set to run at boot via `systemd`.

### 9.3 Phase 3 Setup

1. Connect Analog Discovery 3 to Pi USB.
2. Clamp current probe around +5VSB wire (or connect in‑line with LPC header).
3. Install WaveForms SDK or `pydwf` on Pi.
4. Run calibration: measure baseline current with PC off, then with PC on (ME idle).
5. Deploy detection script.

### 9.4 Phase 4 Setup (Advanced)

1. Program the FPGA with the PCIe sniffer bitstream using the FT2232H programmer.
2. Insert FPGA board into M.2 slot (or riser).
3. Boot target PC; verify the NVMe drive (if behind FPGA) is detected.
4. Run test DMA from PCH (e.g., using `pcileech`) to confirm alerts trigger.

## 10. Testing and Validation

**Phase 1:**  
- Enable AMT in BIOS (if supported) and observe Suricata alert on Pi.  
- Spoof ME MAC address from another machine; verify detection.

**Phase 2:**  
- Use `flashrom` to write a dummy byte to the flash (with write‑enable). `spi_guard.py` should alert.  
- Perform a legitimate BIOS update; alert should **not** fire if system is in update mode (requires whitelisting).

**Phase 3:**  
- Use Intel **AMT WebUI** ping to wake ME; observe 32 kHz harmonic increase.  
- Run a DMA stress test from a PCIe device; observe current signature change.

**Phase 4:**  
- Use **PCILeech** to perform a DMA read from the FPGA’s perspective; confirm packet drop.  
- Boot a Linux kernel with `iommu=pt` and verify that the NVMe drive is still functional.

## 11. Limitations and Future Improvements

**Current Limitations:**
- Phase 3 (Power Oracle) requires calibration per motherboard model.
- Phase 4 (PCIe Arbiter) may reduce NVMe performance (5–10% overhead).
- The system assumes the attacker has not physically tampered with the Sentry hardware itself.
- Windows support for FPGA tools is more complex; Linux is recommended for development.

**Future Enhancements:**
- **Integrated Single‑Board:** Design a custom PCB with RP2040/STM32 for SPI sniffing and a small FPGA for PCIe monitoring, all managed by a Raspberry Pi CM4.
- **Machine Learning on Power Traces:** Deploy TinyML models on the Pi to classify ME activity patterns.
- **Automatic ME Halting:** Use the HECI command `HMRFPO` to disable the ME on alert (requires soldering to LPC bus).

## Conclusion

Building a **hardware‑based anomaly detection system** for the Intel ME / AMD PSP is a complex but achievable project. By layering multiple independent sensors—network, SPI flash, power side‑channel, and PCIe DMA—you can achieve a high degree of assurance that the Management Engine is not being used as a backdoor. This document provides a blueprint for constructing such a system using off‑the‑shelf components and open‑source software.

**Remember:** The ultimate defence is to **remove the ME entirely** using `me_cleaner` or switch to open hardware (RISC‑V). Until that is practical for all users, the Panopticon Sentry offers a powerful, verifiable layer of protection.
